# Node.js Backend

Production guidance for Node.js services. Targets Node 20+ (LTS) and TypeScript 5.x.

## Framework choice — make this deliberately

There are four reasonable defaults in 2026; pick based on the constraint, not vibes:

- **Fastify** — High performance, schema-first via JSON Schema or Zod, plugin architecture, great TypeScript support. Best general-purpose default for production REST APIs. Mature, stable, async-first.
- **Hono** — Tiny, fast, runs on Node / Bun / Deno / Cloudflare Workers / Vercel Edge. Best when you want one codebase that runs in Node and at the edge, or you're targeting a serverless runtime. API is close to Express but typed.
- **Express** — The most common in the wild. Choose only when (a) you're maintaining an existing Express codebase or (b) you specifically need an enormous middleware ecosystem. Async error handling is awkward; performance is meaningfully behind Fastify/Hono.
- **NestJS** — Opinionated, batteries-included, decorator-driven. Choose when you have a large team that wants Java/.NET-style structure (modules, DI, guards, interceptors) and the discipline that comes with it. Heavier; faster to onboard a team to a consistent shape, slower to deviate from the framework's opinions.

For a new general-purpose API: **Fastify**. For edge / multi-runtime: **Hono**. For a large enterprise team: **NestJS**. For a quick script or a legacy fit: **Express**.

The examples below lean Fastify; the principles apply to all four.

## TypeScript setup

Use TypeScript. The "is it worth the friction" debate ended several years ago. Type safety pays for itself within a sprint.

`tsconfig.json` defaults that matter:

```json
{
  "compilerOptions": {
    "target": "ES2022",
    "module": "NodeNext",
    "moduleResolution": "NodeNext",
    "lib": ["ES2022"],
    "outDir": "dist",
    "rootDir": "src",

    "strict": true,
    "noUncheckedIndexedAccess": true,
    "noImplicitOverride": true,
    "exactOptionalPropertyTypes": true,

    "esModuleInterop": true,
    "skipLibCheck": true,
    "forceConsistentCasingInFileNames": true,
    "resolveJsonModule": true,
    "isolatedModules": true,

    "sourceMap": true,
    "declaration": false
  },
  "include": ["src/**/*"]
}
```

`noUncheckedIndexedAccess` is the single most valuable non-default — it forces you to handle `arr[i]` as possibly undefined, which is the truth in JS.

Run TypeScript with `tsx` (or `ts-node` / `node --import tsx`) in dev. In production, build with `tsc` (or `esbuild` / `swc` for speed) and run plain Node on the compiled JS. Don't ship transpilers to production.

For ESM vs CJS: new projects, ESM (`"type": "module"`). Add `.js` extensions in imports (yes, even though you're writing `.ts`) — that's the NodeNext behavior.

## Project structure

```
src/
├── server.ts            # build & start the app
├── app.ts               # framework instance, plugins, route registration
├── config.ts            # env validation via Zod
├── routes/
│   ├── users.ts
│   └── orders.ts
├── services/            # business logic
│   ├── users.ts
│   └── orders.ts
├── db/
│   ├── client.ts        # Drizzle / Prisma client
│   ├── schema.ts        # Drizzle schema (or generated by Prisma)
│   └── migrations/
├── schemas/             # Zod schemas (request/response validation)
├── lib/                 # cross-cutting helpers (logger, errors, jwt)
└── workers/             # background jobs
test/
package.json
tsconfig.json
```

The split that matters: **routes are thin** (parse + validate → call service → return). **Services own business logic** and don't know about HTTP. **Schemas (Zod) own the wire contract.** This makes the service layer trivially testable without a fake request.

## Configuration

Validate environment with Zod at startup. Crash on missing/invalid config — do not lazy-read `process.env` deep in the call stack.

```ts
// src/config.ts
import { z } from 'zod';

const Schema = z.object({
  NODE_ENV: z.enum(['development', 'test', 'production']).default('development'),
  PORT: z.coerce.number().int().positive().default(3000),
  DATABASE_URL: z.string().url(),
  REDIS_URL: z.string().url().optional(),
  JWT_SECRET: z.string().min(32),
  JWT_TTL_SECONDS: z.coerce.number().int().positive().default(3600),
  LOG_LEVEL: z.enum(['fatal', 'error', 'warn', 'info', 'debug', 'trace']).default('info'),
  CORS_ORIGINS: z.string().transform(s => s.split(',').filter(Boolean)).default(''),
});

const parsed = Schema.safeParse(process.env);
if (!parsed.success) {
  console.error('Invalid environment:', z.prettifyError(parsed.error));
  process.exit(1);
}

export const config = parsed.data;
export type Config = z.infer<typeof Schema>;
```

## App and server

```ts
// src/app.ts
import Fastify, { type FastifyInstance } from 'fastify';
import cors from '@fastify/cors';
import helmet from '@fastify/helmet';
import sensible from '@fastify/sensible';
import { ZodTypeProvider, validatorCompiler, serializerCompiler } from 'fastify-type-provider-zod';

import { config } from './config.js';
import { registerErrorHandler } from './lib/errors.js';
import { registerRoutes } from './routes/index.js';

export async function buildApp(): Promise<FastifyInstance> {
  const app = Fastify({
    logger: {
      level: config.LOG_LEVEL,
      ...(config.NODE_ENV === 'development' && {
        transport: { target: 'pino-pretty' },
      }),
    },
    disableRequestLogging: false,
    requestIdHeader: 'x-request-id',
    requestIdLogLabel: 'reqId',
    trustProxy: true,
  }).withTypeProvider<ZodTypeProvider>();

  app.setValidatorCompiler(validatorCompiler);
  app.setSerializerCompiler(serializerCompiler);

  await app.register(helmet);
  await app.register(cors, { origin: config.CORS_ORIGINS, credentials: true });
  await app.register(sensible);

  registerErrorHandler(app);
  await registerRoutes(app);

  return app;
}
```

```ts
// src/server.ts
import { buildApp } from './app.js';
import { config } from './config.js';

const app = await buildApp();

const shutdown = async (signal: string) => {
  app.log.info({ signal }, 'shutting down');
  try {
    await app.close();
    process.exit(0);
  } catch (err) {
    app.log.error({ err }, 'error during shutdown');
    process.exit(1);
  }
};

process.on('SIGTERM', () => shutdown('SIGTERM'));
process.on('SIGINT', () => shutdown('SIGINT'));

await app.listen({ port: config.PORT, host: '0.0.0.0' });
```

`buildApp` returns the app without listening — perfect for tests (`app.inject(...)` instead of opening a port).

## Validation with Zod

Zod is the standard. With `fastify-type-provider-zod`, your route schemas double as TypeScript types — no duplication.

```ts
// src/schemas/user.ts
import { z } from 'zod';

export const UserCreate = z.object({
  email: z.string().email().max(320),
  displayName: z.string().min(1).max(80),
  password: z.string().min(12).max(128),
});
export type UserCreate = z.infer<typeof UserCreate>;

export const UserPublic = z.object({
  id: z.string().uuid(),
  email: z.string().email(),
  displayName: z.string(),
  createdAt: z.string().datetime(),
});
export type UserPublic = z.infer<typeof UserPublic>;
```

```ts
// src/routes/users.ts
import type { FastifyInstance } from 'fastify';
import { z } from 'zod';
import { UserCreate, UserPublic } from '../schemas/user.js';
import { UserService } from '../services/users.js';

export async function userRoutes(app: FastifyInstance) {
  app.post('/users', {
    schema: {
      body: UserCreate,
      response: { 201: UserPublic },
    },
    handler: async (req, reply) => {
      const user = await UserService.create(req.body);
      return reply.code(201).send(user);
    },
  });
}
```

`req.body` is now typed as `UserCreate` automatically. Reject any reflex to add a separate TypeScript interface — Zod is the source of truth.

## Database — Drizzle vs Prisma

Both are good. Pick based on what you want from the ORM:

**Drizzle** — Closer to SQL, schema is TypeScript code, queries return precisely the columns you select, very fast, no separate generation step. Best when you (or the team) like SQL and want minimal abstraction. Migration story is solid (`drizzle-kit`).

**Prisma** — Schema in a `.prisma` DSL, generated typed client, very ergonomic for nested relations and CRUD. Heavier runtime, separate query engine binary. Best when you want maximal ergonomics and don't mind the generation step. Migrations via `prisma migrate`.

For most new projects: **Drizzle**. For teams who've used Prisma productively: stay on Prisma. Don't migrate without a reason.

### Drizzle example

```ts
// src/db/schema.ts
import { pgTable, uuid, varchar, boolean, timestamp, index } from 'drizzle-orm/pg-core';

export const users = pgTable('users', {
  id: uuid('id').primaryKey().defaultRandom(),
  email: varchar('email', { length: 320 }).notNull().unique(),
  hashedPassword: varchar('hashed_password', { length: 255 }).notNull(),
  displayName: varchar('display_name', { length: 80 }).notNull(),
  isActive: boolean('is_active').notNull().default(true),
  createdAt: timestamp('created_at', { withTimezone: true }).notNull().defaultNow(),
}, (t) => ({
  emailIdx: index('users_email_idx').on(t.email),
}));
```

```ts
// src/db/client.ts
import { drizzle } from 'drizzle-orm/node-postgres';
import { Pool } from 'pg';
import { config } from '../config.js';
import * as schema from './schema.js';

const pool = new Pool({
  connectionString: config.DATABASE_URL,
  max: 10,
  idleTimeoutMillis: 30_000,
});

export const db = drizzle(pool, { schema });
export type DB = typeof db;
```

```ts
// src/services/users.ts
import { eq } from 'drizzle-orm';
import { db } from '../db/client.js';
import { users } from '../db/schema.js';
import type { UserCreate } from '../schemas/user.js';
import { hashPassword } from '../lib/security.js';
import { Conflict } from '../lib/errors.js';

export const UserService = {
  async create(data: UserCreate) {
    const [existing] = await db.select({ id: users.id }).from(users).where(eq(users.email, data.email));
    if (existing) throw new Conflict('Email already in use');

    const [user] = await db.insert(users).values({
      email: data.email,
      hashedPassword: await hashPassword(data.password),
      displayName: data.displayName,
    }).returning();

    return toPublic(user);
  },

  async getById(id: string) {
    const [user] = await db.select().from(users).where(eq(users.id, id));
    return user ? toPublic(user) : null;
  },
};

function toPublic(u: typeof users.$inferSelect) {
  return {
    id: u.id, email: u.email, displayName: u.displayName,
    createdAt: u.createdAt.toISOString(),
  };
}
```

Notice: the service returns the *public* shape, not the raw row. Hashed passwords never escape the data layer.

## Authentication

Standard pattern: short-lived access JWT + revocable refresh token.

```ts
// src/lib/security.ts
import * as argon2 from 'argon2';
import jwt from 'jsonwebtoken';
import { config } from '../config.js';

export const hashPassword = (plain: string) => argon2.hash(plain, { type: argon2.argon2id });
export const verifyPassword = (plain: string, hashed: string) => argon2.verify(hashed, plain);

export const signAccessToken = (sub: string) =>
  jwt.sign({ sub }, config.JWT_SECRET, { expiresIn: config.JWT_TTL_SECONDS, algorithm: 'HS256' });

export const verifyAccessToken = (token: string) =>
  jwt.verify(token, config.JWT_SECRET, { algorithms: ['HS256'] }) as { sub: string };
```

For Fastify, an auth hook attached to the routes that need it:

```ts
// src/lib/auth.ts
import type { FastifyRequest, FastifyReply } from 'fastify';
import { verifyAccessToken } from './security.js';

declare module 'fastify' {
  interface FastifyRequest { userId?: string; }
}

export async function requireAuth(req: FastifyRequest, reply: FastifyReply) {
  const header = req.headers.authorization;
  if (!header?.startsWith('Bearer ')) return reply.code(401).send({ error: { code: 'unauthorized' } });
  try {
    const { sub } = verifyAccessToken(header.slice('Bearer '.length));
    req.userId = sub;
  } catch {
    return reply.code(401).send({ error: { code: 'unauthorized' } });
  }
}

// in route:
app.get('/me', { preHandler: requireAuth, schema: { response: { 200: UserPublic } } },
  async (req) => UserService.getById(req.userId!));
```

For browser sessions, prefer server-side sessions (opaque token + Redis) over JWTs in cookies. Easier to revoke, no premature optimization. For social login / SSO, use Auth0, Clerk, WorkOS, or similar — don't implement OAuth2 flows by hand.

## Error handling

The shape that holds up:

```ts
// src/lib/errors.ts
import type { FastifyInstance } from 'fastify';
import { ZodError } from 'zod';

export class AppError extends Error {
  statusCode = 500;
  code = 'internal_error';
  constructor(message: string) { super(message); this.name = this.constructor.name; }
}
export class NotFound extends AppError { statusCode = 404; code = 'not_found'; }
export class Conflict extends AppError { statusCode = 409; code = 'conflict'; }
export class Forbidden extends AppError { statusCode = 403; code = 'forbidden'; }
export class BadRequest extends AppError { statusCode = 400; code = 'bad_request'; }

export function registerErrorHandler(app: FastifyInstance) {
  app.setErrorHandler((err, req, reply) => {
    if (err instanceof ZodError) {
      return reply.code(400).send({
        error: { code: 'validation_error', issues: err.issues },
      });
    }
    if (err instanceof AppError) {
      return reply.code(err.statusCode).send({
        error: { code: err.code, message: err.message },
      });
    }
    req.log.error({ err }, 'unhandled error');
    return reply.code(500).send({ error: { code: 'internal_error' } });
  });
}
```

Services throw `NotFound`, `Conflict`, etc. Routes don't try/catch; the handler does the translation.

**Always handle promise rejections.** An unhandled rejection in modern Node terminates the process. Either every promise has a `.catch()` or it's awaited inside a function whose caller handles errors. Top-level:

```ts
process.on('unhandledRejection', (err) => {
  app.log.fatal({ err }, 'unhandled rejection');
  process.exit(1);
});
process.on('uncaughtException', (err) => {
  app.log.fatal({ err }, 'uncaught exception');
  process.exit(1);
});
```

Crashing on these is correct — the process is in an unknown state; let the supervisor restart you.

## Async patterns — the failure modes

- **Forgetting `await`.** TypeScript catches some via `no-floating-promises` ESLint rule — turn it on.
- **Sequential awaits when parallel works.** `await a; await b;` runs serially. Use `Promise.all([a, b])` when there's no dependency.
- **`Promise.all` failing fast.** One rejection cancels the rest's value. Use `Promise.allSettled` when you want all results regardless.
- **Async in `.forEach`.** `.forEach(async fn)` does not await. Use `for...of` (sequential) or `Promise.all(arr.map(fn))` (parallel).
- **Long-running CPU work in the event loop.** Block other requests. Move to a worker thread (`node:worker_threads`) or a separate process / queue.
- **Streams not handled with `pipeline`.** Use `node:stream/promises` `pipeline` for proper error propagation and cleanup.

## Background work

Tiered, same as everywhere:

1. **Fire-and-forget after response** — fine for unimportant emails, log shipping. Use sparingly; failures vanish silently.
2. **Real queue** — BullMQ (Redis-backed) is the Node standard. Replaces Kue/Bee. Reliable, supports delays, repeats, priorities, rate-limiting. For Postgres-only stacks, `pg-boss` is the equivalent.
3. **Cron / scheduled tasks** — `node-cron` for in-process; for production, prefer external schedulers (Kubernetes CronJob, Cloud Scheduler, BullMQ repeatable jobs).

```ts
// BullMQ producer (in your route)
await emailQueue.add('order-confirmation', { orderId: order.id }, {
  attempts: 5,
  backoff: { type: 'exponential', delay: 5_000 },
});

// BullMQ worker (separate process)
const worker = new Worker('email', async (job) => {
  if (job.name === 'order-confirmation') {
    await sendOrderConfirmation(job.data.orderId);
  }
}, { connection });
```

## Testing

Use **Vitest** (or Node's built-in `node:test` if you're committed to zero deps). `Jest` is fine but slower and heavier; for new projects pick Vitest.

For HTTP tests, prefer `app.inject` over starting a real server — faster, no port allocation, no flake.

```ts
// test/users.test.ts
import { describe, it, expect, beforeAll, afterAll } from 'vitest';
import { buildApp } from '../src/app.js';

let app: Awaited<ReturnType<typeof buildApp>>;

beforeAll(async () => { app = await buildApp(); });
afterAll(async () => { await app.close(); });

describe('users', () => {
  it('creates and fetches a user', async () => {
    const create = await app.inject({
      method: 'POST',
      url: '/users',
      payload: { email: 'a@example.com', displayName: 'A', password: 'x'.repeat(12) },
    });
    expect(create.statusCode).toBe(201);
    const body = create.json();
    expect(body.email).toBe('a@example.com');
  });
});
```

For database isolation: run a real Postgres (testcontainers or a Docker compose `test` profile). Each test runs in a transaction that rolls back, OR each test gets a fresh schema. SQLite-as-fake-Postgres lies in too many ways to recommend.

## Observability

- **Logging: Pino.** It's the Node logger. Structured JSON, fast, integrates with Fastify. Don't use `console.log` in production code.
- **Metrics:** Prometheus via `prom-client`. Expose `/metrics`. Track HTTP latency histograms, error rates, queue depth, business counters.
- **Tracing:** OpenTelemetry with the Fastify and PG/Drizzle/Prisma instrumentations. Auto-traces every request and DB call, correlates them via traceparent.
- **Healthchecks:** `/healthz` (process alive, no DB) and `/readyz` (DB reachable, dependencies OK). Used by load balancers and Kubernetes probes.

```ts
app.get('/healthz', async () => ({ ok: true }));
app.get('/readyz', async () => {
  await db.execute(sql`SELECT 1`);
  return { ok: true };
});
```

## Security defaults

- `@fastify/helmet` — security headers.
- `@fastify/cors` — restricted origins, never `*` for credentialed APIs.
- `@fastify/rate-limit` — per-IP and per-user limits on auth endpoints.
- Validate every input with Zod. Never trust the client.
- Parameterize every query (Drizzle/Prisma do this; raw SQL via tagged templates).
- Hash passwords with argon2id, not bcrypt-with-low-rounds, never plain SHA.
- Set body size limits (Fastify default is 1MB; raise per-route if needed).
- Use `secure`, `httpOnly`, `sameSite=strict` cookies; never put credentials in localStorage.
- Keep dependencies current (`npm audit`, Renovate/Dependabot). Most "Node breach" stories are old packages.

## Deployment

- **Build TypeScript ahead of time.** `tsc -p tsconfig.build.json` (or `tsup`/`esbuild`/`swc`). Run plain Node on the JS output.
- **One process per container; multiple containers for scale.** Don't use `cluster` in Kubernetes — let the orchestrator scale replicas.
- **Graceful shutdown** on SIGTERM. Wait for in-flight requests, drain queues, close DB pools, then exit.
- **Health and readiness probes** wired to `/healthz` and `/readyz`.
- **`NODE_ENV=production`** — many libraries (Express, etc.) take faster paths under it. Forgetting this is the most common Node deployment bug.
- **Memory limits** set at the orchestrator and at the Node level (`--max-old-space-size=NNN`) below the container limit, so V8 GCs aggressively before the OOM killer fires.

## Anti-patterns

- Mixing CommonJS and ESM in one project. Pick one — ESM for new code.
- Catching all errors at the top of every async function. Let them propagate to the framework's error handler.
- Manual JSON parsing instead of using the framework's `request.body`. Skips validation.
- Storing JWTs in `localStorage`. XSS exposure. Use httpOnly cookies for browser auth.
- Returning ORM rows directly. Same reason as Python: contracts and field leakage.
- Reaching for Redis to cache 100 rows. Postgres is fine.
- Putting business logic in middleware. Hard to test, hard to reason about ordering.
- Singleton services with mutable state. Now you have a hidden global, often per-process, broken under any concurrency.
- `setInterval` for "cron." Doesn't survive restarts, no leader election. Use a real scheduler.
- Not testing under load before launch. Most production incidents are first-million-request bugs.
